Per PRD §16.3 ("goldens are minted once per (test × platform-decision ×
app-version), visually inspected by the QA subagent, and committed together
with `BASELINE-APPROVAL.md` ... Re-baselining is a `LENS-TEST-CHANGE` event

minted these goldens in the SAME change that adds `pane.glance`. This file
records that fact honestly rather than asserting an approval that hasn't
happened yet.

## Who / when / why

- **Who:** P2 implementation agent (this branch, `feat/lens-p2-glance`),
  task 077-shux-lens, phase P2.
- **When:** see `evidence-manifest.json` → `minted_at_utc`.
- **Why:** G2/G2w (`crates/shux/tests/lens_glance.rs`) require golden PNG/text
  files at `.shux/goldens/lens/{g2_f1_80x24,g2w_f5_100x30}.{png,txt}` that do
  not exist before P2. They were generated by driving the real daemon through
  the exact steps the frozen tests use (session.create → pane.set_size →
  pane.send_keys exec → pane.wait_for sentinel → `pane.glance{checkpoint:true}`)
  and writing the resulting `text`/`png_base64` fields to disk.

## Known, expected characteristics (not defects)

- Both goldens render CJK/Devanagari script runs as "tofu" replacement-glyph
  boxes. This is a property of the EXISTING, UNCHANGED `shux-raster` bundled
  font chain (JetBrains Mono NF + Noto Math/Symbols/Symbols2/Emoji — no CJK
  or Devanagari font in the bundle), not something introduced by
  `pane.glance`. LENS-R-013 requires reusing shux-raster unchanged, and this
  agent did not touch its font logic. The text side of the golden (the
  `.txt` files) DOES carry the correct Unicode codepoints — only the PIXEL
  rendering falls back to tofu for scripts the bundled fonts don't cover.
